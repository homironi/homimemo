## フィードの投稿取得タイミング

[Fetcherについて](/articles/sanedrgdppjkrqd3oied02se/)でも書いてます。

1日1回Cloudflare WorkersのCron Triggersを使ってポーリング。  
あとはフロントエンドから更新ボタンが押された時に再取得。ポーリング含め、最終取得から1時間経過していなければ再取得は行わないようになっています。つまり操作が無い時は1日1回のポーリング以外は行いません。  
ユーザー数は自分だけと考えるとそこそこリクエスト数を抑えられているかなと思います。

## フィードのパース

[Feedsmith](https://feedsmith.dev/)で基本的に問題なくできました。めっちゃ便利です。

あとはdescriptionにHTMLタグが入っているものがあったので、そのあたりは正規表現で置き換えるようにしたらいい感じになりました。  
HTMLとしてパースしてわっしょいわっしょいしていくのはライブラリを入れるとかCloudflare Workersとの相性とかいろいろ面倒だし、descriptionだけの用途だったので過剰かなということで正規表現での置換にしました。

<Accordion summary="コードメモ">

```ts
const HTML_TAGS = [
  "a", "abbr", "address", "article", "aside",
  "b", "blockquote", "br", "button",
  "caption", "cite", "code", "col", "colgroup",
  "dd", "del", "details", "dfn", "div", "dl", "dt",
  "em",
  "figcaption", "figure", "footer", "form",
  "h1", "h2", "h3", "h4", "h5", "h6", "head", "header", "hr", "html",
  "i", "iframe", "img", "input",
  "kbd",
  "label", "li",
  "main", "mark", "meta",
  "nav",
  "ol",
  "p", "picture", "pre",
  "q",
  "s", "section", "select", "small", "span", "strong", "sub", "summary", "sup",
  "table", "tbody", "td", "textarea", "tfoot", "th", "thead", "time", "tr",
  "u", "ul",
  "var",
  "wbr",
].join("|");

const HTML_TAG_RE = new RegExp(`<\\/?(?:${HTML_TAGS})(?:\\s[^>]*)?\\s*/?>`, "gi");

const NAMED_ENTITIES: Record<string, string> = {
  "&apos;": "'",
  "&mdash;": "—",
  "&ndash;": "–",
  "&hellip;": "…",
  "&laquo;": "«",
  "&raquo;": "»",
  "&lsquo;": "\u2018",
  "&rsquo;": "\u2019",
  "&ldquo;": "\u201C",
  "&rdquo;": "\u201D",
};
const NAMED_ENTITY_RE = new RegExp(
  Object.keys(NAMED_ENTITIES).map(k => k.replace(/&/g, "&").replace(/;/g, ";")).join("|"),
  "g",
);

/**
 * descriptionなどからHTMLタグっぽいテキストを除去する
 * @param html HTMLタグが含まれているかもしれないテキスト
 * @returns HTMLタグっぽいものを除いたテキスト。空文字になったらnull
 */
export function extractDescription(html: string | null | undefined): string | null {
  if (!html) return null;

  const text = html
    .replace(/<br\s*\/?>/gi, " ") // <br>をスペースに（タグ除去前に）
    .replace(HTML_TAG_RE, "")
    .replace(/&nbsp;/g, " ")
    .replace(/&amp;/g, "&")
    .replace(/&lt;/g, "<")
    .replace(/&gt;/g, ">")
    .replace(/&quot;/g, '"')
    .replace(NAMED_ENTITY_RE, match => NAMED_ENTITIES[match] ?? match)
    .replace(/&#(?:x[\da-f]+|\d+);/gi, decodeNumericEntity)
    .replace(/[\u200B-\u200D\uFEFF]/g, "") // ゼロ幅文字を除去（\s+より前に）
    // eslint-disable-next-line no-irregular-whitespace
    .replace(/　/g, " ") // 全角スペース
    .replace(/\s+/g, " ")
    .trim();

  return text || null;
}

function decodeNumericEntity(entity: string): string {
  const match = entity.match(/&#(x[\da-f]+|\d+);/i);
  if (!match) return entity;
  const code = match[1].startsWith("x") || match[1].startsWith("X")
    ? parseInt(match[1].slice(1), 16)
    : parseInt(match[1], 10);
  return String.fromCodePoint(code);
}
```

## 定期実行

Cloudflare WorkersのCron Triggersで1日1回以下の内容を実行します。

- 誰も購読していないフィードデータの削除
- 保存期間（32日間）を過ぎた投稿データの削除
- フィードの取得

## 認証

Better AuthのGoogleアカウントログインとafterのhookでアドレス確認→自分のアドレス以外は削除（Googleログインなので事前にとれないので後から）という感じにしています。  
さらにそれより前にフィードリーダー全体にCloudflare Accessでの認証もかけています。こちらも自分のメールアドレスのログインルールを入れています。

Cloudflare Access → Better Auth のGoogleログインの2段階です。

Cloudflare Accessのおかげもあってbotも弾けてます。  
最初はbotがphpファイルとかを探しに来てましたが、HonoのAPIなども含めすべて認証必須にしたうえでCloudflare Accessの奥に隠し、そもそもphpとか使っていない技術にアクセスしてくるものはすべて悪いやつというルールを設定したら0になりました。  
Cloudflareがバンバン弾いてくれてます。

## 未対応だけどやりたいこと

### WebSub対応

WebSubの存在自体知らなかったんですが、ポーリングを最低限にできそうでいいなーと思ったのでやりたいです。  
旧称は`PubSubHubbub（パブサブハブバブ）`です。ふざけてるのかなと思ってしまう文字の並びにいつもニコニコと文字列を見ています。こういう文字列がめっちゃすき。

まずはこのサイトのフィードをWebSub（[Google PubSubHubbub Hub](https://pubsubhubbub.appspot.com/)を使うと厳密にはPubSubHubbubなんだろうか）に対応させて、それを使いながら実装してみようと思っています。

<TextBlock blockType="note" title="もにょもにょ">
WebSubと検索すると不要論がでてきて自分としてはちょっとしょんもり。フィード配信にはめっちゃいいと思うので全く不要であるというのは（SEOの観点でしかみてないと思うので仕方ないと思いつつ）なんだかもにょもにょ……と思いました。  
どうやら昔はGoogleのインデックスが早くなるみたいなそういうのがあったらしいですね。
</TextBlock >

実際、フィード自体がおそらくユーザーが減っていてそのフィードのためにWebSub対応とかは過剰なんだろうなあ、と思っています。  
自分もフィードリーダーで使いたいから実装してみようかな、というだけでそれがなければこのサイトのフィードでWebSub対応は過剰なので「やりたい」以外の理由ではやらない気がします。

<TextBlock blockType="note" title="WebSub">
WebSubはフィード配信元（publishers：多分Pubの部分）、フィードの購読者（subscribers：多分Subの部分。ポーリングするやつ）、配信元と購読者をつなぐHubが出てきます。  
本来はSubscriber→Publisherにポーリングしますが、WebSub対応されているとポーリングが減らせます。ポーリングが減るということはフィードへのアクセスが減る→サイトのアクセスが減る→負荷がちょいさがる、というわけです。  
Subscriber→PublisherだったところにHub（[Google PubSubHubbub Hub](https://pubsubhubbub.appspot.com/)など）が入り中継役をしてくれます。  
Publisherは事前にHubに登録し、記事の公開などフィード更新時に更新をHubに通知。  
Hubは受け取った通知をSubscriberに送ります。  
Push型のテクノロジーとか呼ばれていることもあるみたいです。
</TextBlock >

例えば毎時ポーリングから、1日1回ポーリングで後はHubからの更新を受け取るだけ、みたいな感じにすることができます。HomiFeedではWebSub対応をしたら、対応サイトは1日1回ポーリングとHubからの更新通知のみにしようかなと思っています。  
現状は1日1回ポーリングと最低1時間でUIから更新リクエストが飛ぶまではポーリングしない、という感じなのでそこまで劇的に減るわけではないですが、やってみたいので気が向いたらやります。

### 言語フィルタリング

これはミュートワードよりもさらに気が向いたらのやつです。

英語のフィードがあるのですが、翻訳かけちゃいたいけど日本語と混ざっていると日本語が変に翻訳されて情報がちょっと減ることがあります。  
これを言語で一時的にフィルタリングできたら英語のフィードだけ翻訳かけてみる、がしやすいなあと。  
フォルダ分けもしてますが言語でもフォルダ分けするとなんか分けすぎてえらいことになってしまうのでヌヌヌです。  
フォルダに加えてフィルタリングできるタグという機能を入れて代用するのもありかなあと思いつつ、そもそも別にまあいっかと思っているので今はメモだけしといて気が向いたら、という感じです。
